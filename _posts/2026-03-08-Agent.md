LLM 智能体核心体系笔记（极简结构化版）
核心逻辑：以「大模型仅能文本补全」为底层前提，所有组件围绕「让大模型具备工具能力、解决实际问题」构建，核心分为4大基础层，其余均为层内组件/能力。
一、核心四要素（体系基石）
这是整个架构的“骨架”，所有概念均依附于此，对应你总结的核心分类。
要素名称 核心定位 本质/能力 一句话口诀 
LLM（大语言模型） 核心大脑 纯文本补全工具，无自主工具调用、无实时知识 只会“填字”，不懂“做事” 
Prompt 体系 沟通语言 大模型的输入指令，包含上下文、记忆、任务要求 给大脑“下指令+给素材” 
Agent（智能体） 执行中枢 监听/解析大模型输出，调度工具、执行任务 大脑的“手脚+翻译官” 
MCP（模型上下文协议） 通信标准 统一 Agent 与外部工具的对接格式，简化多工具适配 工具对接的“通用接口” 
二、Prompt 体系（层内组件）
所有组件均为「优化 Prompt 生成与管理」，是大模型的“输入优化器”，你可理解为“不同封装程度的提示词工具”。
组件名称 核心功能 与 Prompt 的关系 实操价值 
Prompt（提示词） 基础指令单元 核心本体，直接定义任务、约束、要求 最基础，所有组件的起点 
Context（上下文） 对话语境承载 Prompt 的核心组成部分，包含当前对话信息 让大模型“知道当下聊什么” 
Memory（记忆） 历史信息管理 属于 Context 的扩展，存储历史对话/长期信息 让大模型“记住之前聊什么” 
Skill（技能） 可复用任务封装 高度封装的「Prompt 模板 + 固定流程」 一键调用（如“整理日报”），避免重复写提示词 
LangChain 复杂任务编排框架 拆分任务为多步 Prompt，串联工具调用 图形化/代码化“组装提示词”，实现多步骤任务 
Workflow（工作流） 任务执行顺序定义 与 LangChain 核心逻辑一致，侧重“步骤可视化” 明确“先检索、再总结、最后生成”的执行链路 
三、Agent 体系（层内能力/组件）
Agent 是“执行中枢”，以下是其核心能力与可调度的具体工具，是大模型接入电子世界的核心载体。
（一）Agent 核心能力
能力名称 核心作用 底层逻辑 
Function Calling（工具调用） 解析大模型的“工具指令” 大模型按指定格式输出 JSON（暗号），Agent 识别并执行 
监听/解析 核心基础能力 持续接收 LLM 输出，区分“自然语言回答”和“工具调用指令” 
调度/执行 核心执行能力 按解析结果，调用对应工具，处理执行结果 
（二）Agent 可调度的核心工具（Skill 具体落地）
工具名称 核心功能 执行流程 
RAG（检索增强生成） 解决知识过时/幻觉问题 1. 问题向量化；2. 向量库检索相似片段；3. 拼接至 Prompt；4. 大模型基于片段生成回答 
Search（搜索） 获取实时/公开知识 Agent 调用搜索接口，将结果回填至 Prompt，交由 LLM 生成回答 
SubAgent（子智能体） 拆分复杂子任务 主 Agent 调度子 Agent 处理细分任务（如“先写代码、再查bug”） 
四、MCP 协议（独立补充）
核心属性 具体说明 解决的痛点 
本质 统一的工具通信格式标准 避免 Agent 为每个工具单独开发对接代码 
作用对象 Agent ↔ 外部工具（API、数据库、本地程序） 实现“一次适配，多工具通用” 
与 Function Calling 区别 Function Calling 是 LLM→Agent 的指令格式；MCP 是 Agent→工具的通信格式 前者管“大脑给手脚下指令”，后者管“手脚和外部工具说话” 
五、体系协作总流程（闭环记忆）
1. 用户输入：提出问题（如“查2026年最新机器人行业就业数据”）。
2. Prompt 构建：结合 Context/Memory，生成基础指令。
3. LLM 输出：识别需工具调用，按 Function Calling 格式输出 JSON 暗号。
4. Agent 解析：监听并识别暗号，确定需调用「Search」工具。
5. MCP 适配：Agent 按 MCP 协议，对接搜索工具的统一接口。
6. 工具执行：搜索工具返回最新就业数据。
7. 结果回填：Agent 将数据拼接成新的 Prompt（含参考资料），喂回 LLM。
8. 最终生成：LLM 基于事实数据，输出自然语言回答。
